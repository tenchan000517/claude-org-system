# 部署（多領域モデル）

1人/1領域を超えた組織向け。部署とは**自分の説明責任を持つスコープされた領域**。
1人運用はこれを丸ごと省略——作業員空間1つとリーダーがあればよい。

## 部署＝スコープされた領域

各部署は：

- 1つの**領域**（一貫した作業範囲）を担い、
- 成果に責任を負う**リード**を持ち、
- 自分の**ディレクトリ/空間**で作業を走らせ（文脈と memory がそこにスコープされる）、
- クレドから**働き方ドキュメント**を派生し（`templates/department.CLAUDE.md`）、
- **状態**を組織の地図へ出す。

## 目標：KGI / KPI（または単に「完了の定義」）

部署は成功の意味を名指す：

- **KGI** — 担う目標レベルの成果（「勝ってるか？」の数/状態）。
- **KPI** — 週次で管理する先行指標。

小規模/内部部署は、これを素朴な*完了の定義*に置き換えてよい。要点は、各部署が
「自分は何に説明責任を負い、当たっているとどう分かるか」に答えられること。

## profit vs cost センター（有用な汎用区分）

部署は組織の最上位目標との関係で異なる：

- **profit / impact センター** — 組織の看板成果を直接動かす。
- **cost / enabling センター** — 直接は動かさないが、動かす側を支える（インフラ・支援・内部ツール）。

これを名指すと2つの誤りを防げる：enabling 部署を動かせない看板数字で評価すること、
impact 部署が数字を動かさない活動の陰に隠れること。

## 部署間オーケストレーション

- **組織レベルの地図**（最上位の INDEX）が部署・領域・リード・状態を載せる——リーダーの案件地図と同じパターンを1段上で。
- 部署をまたぐ作業はそのレベルで調整し、決定を決議として記録する（`framework/08-governance.md`）——1部署の memory に埋めない。
- 共有インフラ/情報は**秘書**に住み、各部署が再導出しないようにする。

## 部署の境界（自分の領分に留まる）

部署は自分の判断で他部署の領域へさまよわない。領域横断のアイデアは組織レベルで提案として上げる——
別の明示的な取り組みであって、自己承認の拡張ではない。これはクレドの「最小解釈」を組織スコープに適用したものです。
