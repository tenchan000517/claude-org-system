# memory モデル

システムの心臓。振り分けを正せば、山は二度と形成されない。

## ディレクトリ単位のスコープが実現装置

エージェントのランタイムは通常、作業ディレクトリごとに別々の memory を持つ。だから*どこで開くか*が
*どの memory が読まれるか*を決める。各案件をその専用ディレクトリで作業すれば、memory は自動で分かれる
——新規作業に手動の仕分けは要らない。

## 振り分け規則 — 各種の知識はどこに住むか

| 知識の種類 | 家 |
|---|---|
| 案件固有の状態・決定・作業メモ | その**作業員**ディレクトリの memory |
| インフラ/参照事実（パス・アカウント・設定・常設参照） | **秘書** |
| 名簿・横断的な決定 | **リーダー** |
| 普遍的な働き方ルール（「事実は必ずツールで検証」、ハウススタイル） | ランタイムの**グローバル**設定 |

最重要の一行：**案件の知識はリーダーに落ち着かない。** リーダーで*動いて*よいが、得た知識は作業員へ振り分ける。

## memory の種類（掃除はそれぞれをどう扱うか）

- **働き方 / feedback** — 蓄積された「我々のやり方」。*機能*でありノイズではない。掃除で消さない。普遍なら、全ディレクトリが継承するようグローバル設定へ昇格。
- **参照（reference）** — 引く事実。**秘書**に属す。掃除で消さない。秘書への移行は削除ではなく正しい家への移設。
- **案件（project）** — 進行中作業の状態。作業員に住む。掃除が刈るのは*これ*——移管・完了・陳腐化したとき。
- **ユーザー / 組織事実** — 誰が/何の恒久事実。残す。めったに変わらない。

## 掃除（denoise）— 山を安全に痩せさせる

catch-all memory を痩せさせるとき：

1. まず memory ディレクトリ全体をバックアップ。
2. 働き方型と参照型は自動で残す（保護）。**案件**型だけが候補。
3. 案件エントリが削除候補になるのは**明確に死んでいる**ときだけ：他所へ移管済み（本当の家が別ディレクトリ）・再利用価値なく完了/解決済み・完了タスクのログ・完全重複。それ以外は生きている。
4. 迷ったら残す。
5. 削除は可逆（バックアップ）；1ファイルだけ戻すのはコピーで済む——全部を戻す必要はない。

## 整然さを超えた意味

肥大した混在 memory はセッションを遅くするだけでなく、*混乱の原料*になる：エージェントが古い常設タスクのメモを
ライブな指示と取り違えうる。軽く振り分けられた memory は、その燃料を取り除く。
