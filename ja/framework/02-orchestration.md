# オーケストレーション

catch-all を再生せずに、作業を役割の間に流す方法。

## INDEX が「何が・どこに」の唯一の真実

リーダーの `INDEX` が地図。全ライブ案件を載せる：名前・場所（作業員ディレクトリ）・状態・次の一手。
横断的な決定も載せる。個別案件の作業詳細は載せない——それは作業員に住む。

正直に保つ：休眠/完了案件は「非アクティブ」節へ移すか外に出し、地図が*今ライブな物*を映すようにする。

## 振り分けループ

1. **リーダー**を開く。地図が見える。
2. 案件を選ぶ。リーダーがその**作業員**ディレクトリを指す。
3. **その作業員ディレクトリで**エージェントを開き、そこで作業する。
4. 状態が変わったら**1行更新**が `INDEX` に戻る。
5. 途中で要る参照（パス・アカウント・設定・用語）は**秘書**に尋ねる。

軽い/横断的な作業はリーダーで起きてよいが、*案件固有の出力は作業員の memory へ振り分ける*
（`03-memory-model.md` 参照）。

## セッション間の引き継ぎ

後で再開する作業員（やリーダー）は、短い **HANDOFF** を残す：現在のターゲット・状態・期限/ブロッカー・
次の具体的な一手。次のセッションはまず HANDOFF と INDEX を読み、即座に把握する——山から文脈を再導出しない。

## 上への報告（任意・大組織）

経営/領域リード層がある場合、定期報告が固定フォーマットで上へ転がる
（`templates/reports/monthly-report.md`）。1人運用は省略——INDEX ＋ HANDOFF が既に「現在地」を運ぶ。

## オーケストレーションが決してなってはいけない物

- 案件 memory を溜める場所（＝ catch-all の回帰）。
- 全ての場内作業を禁じるボトルネック（「リーダーは一切作業しない」は1人では非現実的）。

均衡：**知識がどこに落ち着くかで振り分け、労働がどこで起きるかは柔軟に。**
