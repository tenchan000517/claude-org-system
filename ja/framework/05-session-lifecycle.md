# セッションのライフサイクル

1セッションがどう開き・走り・閉じるか。次のセッションがクリーンな状態を継承するように。

## 起動時 — 把握する。再導出しない。

- **リーダー**で：リーダー宣言と `INDEX`（と `HANDOFF` があれば）を読む。何がライブでどこにあるか分かる。
- **作業員**で：その作業員の宣言・`HANDOFF`（現ターゲット/状態/ブロッカー）・自分の memory を読む。この1案件にスコープされる。
- インフラ/参照の問いは、事実を再構築せず**秘書**に尋ねる。

## セッション中 — 進めながら振り分ける

- 正しいディレクトリで作業する。
- 新しい案件知識 → 作業員の memory に落ち着く。学んだ infra 事実 → 秘書へ。
- あらゆる構造変更に安全規則を適用（バックアップ・スコープ・追加的・報告＋復元）。

## 終了時 — 開いたときより把握しやすく残す

1. **worklog** に短い時刻付き行を追記（何が起きたか）。
2. **HANDOFF** を更新：現ターゲット・状態・期限/ブロッカー・次の具体的な一手。
3. 新しい恒久事実/決定を捕捉用に出す（**ingest** 雛形参照）——途中で肥大させず、秘書/memory を最新に保つ。

## 肥大なき事実の捕捉（ingest）

タスク中に KB を編集する代わりに、新しい事実/決定を **ingest キュー**（小さな日付付きファイル）に落とし、
後で意図的に秘書/memory へ畳む。作業セッションを集中させ、KB を堆積でなくキュレートされた状態に保つ。
（任意；1人/小規模は、小さな変更なら KB を直接編集してよい。）

## 要点

各セッションは数秒で把握できて始まり（地図＋handoff を読む・窓口に尋ねる）、地図と handoff を正確にして終わる。
どのセッションも、現在地を知るために山を掘り返さずに済む。
