# KB システムとしての秘書

秘書は組織の**共有インフラ＆情報窓口**。最小形は1つの `KB.md`。大きくなると、次の部品を持つ小さな
ナレッジベースになる：

```
secretary/
  KB.md                 単票形式（1人/小規模）— facts, decisions, references, glossary
  kb-yaml/              構造化形式（1ファイルを超えたら）
    facts.yaml          設定・パス・アカウント・設定値
    decisions.yaml      決議録＋なぜ
    glossary.yaml       名前・別名・対応
  rag-<topic>.md        繰り返し尋ねられるトピックごとに1枚（そのトピックへの習熟）
  ingest/               セッション中に新事実を落とし、後で意図的に畳む
```

## 何を担うか

- **インフラ事実** — 「どこにXが/設定は/どのアカウント」という、各役割が再導出してしまう共有知識。これが**インフラ共有**レイヤ：一度定義し、皆ここで尋ねる。
- **決議録** — 決着した問いを蒸し返さないため（作業レベルの決議ログのミラー）。
- **用語集** — 物ごとに正式名を1つ。
- **RAG ページ** — より深い反復トピックは、要点・手順・落とし穴を載せた専用ページを持つ。

## 各役割の使い方

- 作業員とリードは infra 事実を自分の memory に抱えず、秘書に**尋ねる**。
- 参照/infra 知識は**ここに住む**——案件 memory に散っているなら移し込む（消さず移動・先にバックアップ）。「reference を守る」は*削除しない*であって、家へ*移設しない*ではない。

## ingest 規律（途中の肥大を避ける）

セッション中は新事実/決定を `ingest/` に落とし、後で意図的に KB へ畳む。作業セッションを集中させ、
KB を堆積でなくキュレートされた状態に保つ。1人/小規模は小さな変更なら直接 KB を編集してよい。

## Q&A の姿勢

秘書は**答える**ために開く——案件作業のためではない。窓口：事実とポインタを返す。案件作業は作業員ディレクトリで起きる。
